# vc-decorate -- Coherence First. Premium Second.

> "Do not decorate chaos. First make the system coherent. Then make it feel premium."

Decorate is **not** a "make it pretty" skill.

Decorate is a **late-stage product finishing skill**.

Its job is to take a working product and turn it into a **coherent, intentional, premium experience**.

That means:

- detecting the user's real design language
- separating identity from drift
- preserving what is distinctive
- upgrading what is weak, dated, or inconsistent
- verifying the end-to-end feel of the product
- and **only then** adding tasteful visual polish and micro-interactions

Decorate does **not** impose the agent's taste.
Decorate does **not** overwrite the user's brand.
Decorate does **not** add random blur, glow, parallax, or "AI prettiness."

Its job is to make the existing system feel:

- more deliberate
- more modern
- more stable
- more precise
- more complete

**Premium is not ornament. Premium is coherence.**

---

## Core Rule: Detect, Don't Dictate

Before decorating anything, run style detection and system audit.

```text
1. SCAN existing CSS variables, theme files, brand colors, fonts, spacing, and component patterns
2. IDENTIFY the user's palette, font stack, theme mode, surface logic, and interaction rhythm
3. AUDIT for visual drift, weak patterns, inconsistent states, and prototype-feel leftovers
4. SEPARATE identity from drift:
   - preserve what is distinctive
   - improve what is weak, stale, or incoherent
5. PROPOSE improvements using THEIR tokens, THEIR language, THEIR stack
6. ASK which changes should be applied
7. IMPLEMENT only approved changes
8. VERIFY the experience end-to-end
````

If no existing style is detected, offer to scaffold a minimal design system first --
but present options, don't assume taste, and don't force a visual identity.

---

## What Decorate Is For

Use Decorate when:

* the product works but still feels flat, prototype-ish, or unfinished
* the user asks for visual polish, smaczki, curb appeal, or premium feel
* the UI is functionally correct but lacks coherence across surfaces
* the product has good ingredients but weak system feel
* there are inconsistent cards, buttons, spacing rules, focus states, or animation timings
* a showcase page, demo, landing page, or app needs a finishing pass
* the team wants the product to feel more intentional, not just more decorated

In the VibeCraft pipeline, Decorate belongs **after clarity and before packaging**.

## The Philosophy of Decorate

Decorate is not about your taste.

Decorate is not about making the page louder.

Decorate is not about proving that the agent knows modern CSS tricks.

Decorate is about **respecting the user's design language** while raising the product's level of coherence and feel.

A good decoration pass should make the product feel:

* more intentional
* more trustworthy
* more stable
* more polished
* more unified

The best smaczek is not the most visible one.
It is the one that disappears into the product and makes everything feel more complete.

A good Decorate pass should make the user say:

> "Yes, this still feels like *my* product -- just much more finished."

---

## Decorate Is Not

Decorate is **not**:

* random animation
* visual noise
* glow sprayed over weak layouts
* preserving bad patterns forever just because they already exist
* trend-chasing
* unsolicited rebranding
* converting everything into the agent's favorite aesthetic
* making things busier to make them feel "premium"
* a replacement for product clarity or UX logic

If the product is inconsistent, weak, or chaotic, your first move is not polish.
Your first move is coherence.

---

## Identity vs Drift

One of Decorate's most important jobs is to tell the difference between:

### Identity

The user's actual visual language:

* chosen palette
* real typography decisions
* meaningful spacing rhythm
* distinctive component forms
* product character
* intentional interaction style

### Drift

Things that merely accumulated:

* inconsistent border radii
* mismatched spacing between sections
* two or three conflicting button styles
* random hover behaviors
* shadows that vary for no reason
* stale "default framework" leftovers
* one-off UI treatments that feel accidental
* prototype artifacts that survived too long

Decorate should preserve **identity** and reduce **drift**.

Do not protect weak patterns just because they already exist.
Respect the product's language. Do not worship its inconsistencies.

---

## What Decorate Actually Does

Decorate has five jobs.

### 1. Detect the system

Find the real structure of the visual language:

* color tokens
* typography stack
* theme mode
* spacing rhythm
* border radius logic
* elevation/shadow language
* interaction timing
* hover/focus/active states
* surface hierarchy
* animation style
* current component families

### 2. Audit coherence

Inspect the product for:

* visual drift
* outdated patterns
* inconsistent interaction rules
* broken hierarchy
* accidental layout tension
* too many competing accents
* prototype-feel leftovers
* weak perceived quality

### 3. Propose upgrades

Propose improvements that:

* preserve identity
* strengthen consistency
* replace weak patterns with stronger stack-native solutions
* improve perceived quality
* add polish only where it reinforces the system

### 4. Verify the experience

Review the product end to end:

* hero
* cards
* forms
* nav
* panels
* empty states
* loading states
* focus states
* responsive behavior
* page rhythm
* transitions between surfaces

Decorate should verify whether the product feels like **one product**, not many disconnected screens.

### 5. Apply approved polish

Only after coherence and approval:

* micro-interactions
* hover refinements
* focus refinements
* better timing curves
* subtle motion
* controlled parallax
* better reveal logic
* refined tactile behavior
* premium finish layer

Premium does **not** mean:

* more blur
* more glow
* more gradient
* more motion
* more 3D
* more fancy effects

Premium means:

* coherent spacing
* clean hierarchy
* confidence in surface treatment
* consistent interaction logic
* better restraint
* smoother transitions
* calmer but stronger rhythm
* clear tactile feedback
* visual decisions that feel deliberate

A premium product does not feel *decorated*.

## When To Refuse or Slow Down

Do **not** rush into smaczki when any of the following are true:

* the layout is still structurally broken
* hierarchy is unclear
* screens are not yet product-coherent
* buttons, cards, and text styles are still fighting each other
* the product has no stable visual language yet
* responsiveness is unstable
* basic states are missing
* the user has not approved the direction

In these cases, Decorate should say:

> "Before polish, this needs a coherence pass."

That is not being difficult.
That is being correct.

## System Audit Reference

Before proposing polish, inspect these layers.

### Layout Rhythm

* section spacing
* vertical pacing
* grid consistency
* card widths vs copy density
* line length and readability
* alignment logic
* breathing room around dense sections

### Typography

* heading hierarchy
* body readability
* excessive line length
* inconsistent casing rules
* overly academic vs overly theatrical tone
* mismatch between hero tone and body tone

### Color and Surface Logic

* token usage consistency
* accent overuse
* bg/surface contrast
* border logic
* shadow/elevation logic
* dark mode coherence
* interactive state color behavior

### Components

* button families
* cards
* nav bars
* tabs
* accordions
* chips/tags
* tables / metrics / charts
* panels and sidebars

### Motion and Interaction

* hover behavior consistency
* focus ring logic
* active state feel
* animation timing curves
* page reveal rhythm
* loading states
* number animation / graph labeling readability
* motion density vs product seriousness

### Experience Integrity

* does the product feel like one system?
* does the landing page match the app feel?
* do polished areas expose the roughness of untouched areas?
* does visual confidence hold across sections?

## Style Detection Reference

How to detect and extract the user's design language:

```text
CSS Variables:     grep for --primary, --accent, --bg, --text, --font
Tailwind Config:   read tailwind.config.js for theme.extend.colors
CSS Files:         extract most-used colors (hex/rgb/hsl)
HTML:              inspect loaded fonts, structure, repeated card/button patterns
Package.json:      check for UI frameworks (shadcn, MUI, Chakra, Tailwind, etc.)
Existing Hover:    inspect current :hover, :focus, :active rules
Theme Mode:        check for prefers-color-scheme or .dark class
Animation Layer:   inspect transitions, reveals, keyframes, intersection observers
```

Use the detected values as your working system.

Map them into roles:

* primary/accent = CTA and emphasis logic
* secondary = subtle reinforcement and edge transitions
* background = mood and surface hierarchy
* text = readability and information authority
* motion = energy level and product confidence

## Proposal Model

Decorate should not silently apply a makeover.

Decorate should produce a proposal.

A strong Decorate proposal contains:

1. **What is already working**
2. **What feels weak or inconsistent**
3. **What should be preserved**
4. **What should be upgraded**
5. **What polish options are available**
6. **Which changes are coherence fixes vs optional smaczki**
7. **What the user should approve before implementation**

That means the proposal is not only:

* "Here are 5 hover effects"

It is also:

* "These two button styles should be unified"
* "This card spacing is too cramped for the amount of copy"
* "This heading rhythm is premium in one section and prototype-ish in another"
* "This progress graph needs readability before animation"
* "This navbar layout is fighting the page shell"

---

## Decorate Proposal Categories

### Category 1: Coherence Fixes (highest priority)

These are not optional beauty features.
They are system integrity fixes.

Examples:

* unify card spacing and padding
* normalize heading hierarchy
* fix line length on dense cards
* remove awkward word wrapping / widows / orphan lines
* align top bar width with page shell
* improve label contrast on charts or diagrams
* unify border radii across card families
* reduce conflicting shadow treatments
* fix inconsistent button states
* standardize hover/focus timing

### Category 2: Premium Surface Upgrades

Examples:

* better elevation logic
* more confident panel surfaces
* sharper information hierarchy
* stronger hover/focus states using existing tokens
* improved CTA tactility
* cleaner data visualization labeling
* better empty / loading states
* more polished navbar and shell logic

### Category 3: Smaczki / Micro-Interactions

Examples:

* scale + lift on interactive elements
* subtle glow on hover using the user's accent color
* focus ring improvements
* scroll progress bar
* fade-up on scroll
* parallax only when it matches the product tone
* animated counters
* polished copy interactions
* subtle reveal choreography
* tasteful magnetic or tactile behavior

### Category 4: Agent's Choice

The agent should also propose one enhancement that fits the product's character specifically.

Examples:

* a dashboard could get subtle metric activation behavior
* docs could get improved code copy feedback
* a showcase page could get restrained hero motion
* a data-heavy product could get chart emphasis states
* a design-heavy product could get depth/hover tension based on its actual palette

The best proposed smaczek is:

* not random
* not trendy for its own sake
* but deeply native to the product

## Upgrade Weak Patterns

This is a critical Decorate responsibility.

If the product currently contains weak patterns such as:

* oversized glow buttons
* muddy gradients
* uneven card spacing
* busy shadows
* stale bootstrap-y hover states
* random transitions
* accidental center alignment
* cramped metric cards
* over-rendered hero decoration
* interaction inconsistency between sections

Decorate should **propose upgrades**, not preserve them blindly.

Examples of upgrade logic:

* replace muddy visual noise with cleaner token-based emphasis
* replace inconsistent shadow soup with one elevation system
* replace "more effects" with "better hierarchy"
* replace stale framework defaults with stack-native patterns
* replace unclear motion with one coherent motion language

## End-to-End Experience Review

Decorate should always think beyond isolated components.

Walk through the product like a user would.

Review:

* the landing page
* section rhythm
* transitions between blocks
* nav shell
* hero → evidence → framework → footer flow
* internal app surfaces
* state changes
* loading moments
* edge cases
* responsiveness
* accessibility signals
* performance implications of decoration

Ask:

* does the product feel like one product?
* does the first impression match the real usage surfaces?
* does polish disappear in use, or does it scream for attention?
* does the product feel resolved?

## Performance and Accessibility Guardrails

Decorate is not allowed to create visual debt.

If proposing motion or interaction polish:

* use `requestAnimationFrame` where appropriate
* use passive listeners on scroll
* use `will-change` carefully, not everywhere
* support `prefers-reduced-motion`
* keep focus states accessible
* never destroy keyboard navigation
* never lower contrast for the sake of aesthetics
* do not create fragile animation dependencies
* do not make charts, labels, or metrics harder to read

A premium product that becomes less usable is not more premium.
It is simply worse.

---

## Fallback Design Language (only when truly needed)

When no meaningful design language exists and the user explicitly asks for a starter direction, Decorate may scaffold a minimal system.

But even then:

* present it as a proposal
* explain the rationale
* do not silently impose taste
* keep it restrained and stack-native

Any fallback must be:

* minimal
* calm
* modern
* coherent
* easy to evolve

Fallback is not an excuse to insert the agent's favorite aesthetic.

## Implementation Pattern

```text
Step 1: Detect
  - scan tokens, stylesheets, framework config, component patterns
  - extract palette, fonts, surface logic, spacing, and motion language

Step 2: Audit
  - identify identity vs drift
  - identify weak patterns
  - identify coherence problems
  - identify what already feels strong

Step 3: Propose
  - present coherence fixes first
  - then premium upgrades
  - then optional smaczki
  - include one agent's-choice proposal
  - ask user what to apply

Step 4: Implement
  - apply only approved changes
  - use user's tokens and structure
  - prefer additive changes over destructive rewrites
  - improve weak patterns without erasing identity

Step 5: Verify
  - review before/after
  - check end-to-end experience integrity
  - confirm no user style was betrayed
  - confirm the product feels more finished, not more decorated
```

## Decoration Checklist

Before marking Decorate complete:

```text
[ ] Existing design language detected and understood
[ ] Identity clearly separated from drift
[ ] Weak patterns identified and, where approved, upgraded
[ ] User's style preserved (no silent rebranding)
[ ] Enhancements use the user's palette and tokens
[ ] user-select: none on UI chrome, text on copyable content
[ ] Layout rhythm reviewed
[ ] Typography rhythm reviewed
[ ] Responsive behavior maintained
[ ] Accessibility preserved or improved
[ ] Reduced-motion support respected
[ ] Scroll listeners are passive where used
[ ] requestAnimationFrame used where applicable
[ ] No gratuitous effects added
[ ] End-to-end experience reviewed
[ ] Product feels more coherent, not just more animated
```

---

## Anti-Patterns

Avoid these failure modes:

* decorating a broken structure
* keeping bad patterns because "the user already had them"
* replacing their style with ours
* adding motion without interaction purpose
* adding blur/glow because "premium"
* touching every component instead of fixing the important ones
* over-polishing one surface while the rest stays incoherent
* making the product feel more like a demo and less like a product
* chasing trend aesthetics over system integrity
