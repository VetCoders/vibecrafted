## The **80/20** Rule and **The Definition of Undone**

݆ ݆𝚅𝚒𝚋𝚎𝚌𝚛𝚊𝚏𝚝𝚎𝚍. is a convergence framework for AI-native software development.

It does not promise that AI "will write your code" as this is obvious it does.
It promises something harder and much more valuable: that the code generated by
AI coding agents will be pushed through a loop of finding errors, fixing them,
and verifying them again and again until it is truly ready for release.

Many methods and tools leave you with a first draft.
Some can deliver half of the job.
There are brilliant set of tools so overly complex that they
are out of the reach for non-programmers but would bring you to your first
professional deployement.

This is sometimes up to **80/100** of the job.

## The **20/80** Rule and the conflict markers to resolve

At this stage the proportions revert and you discover that the 80 percent you have is 20 percent of effort you've paid.

## **"What is still wrong?"**

This question boosts the entire system.

First, the agent receives the repository structure, the history of decisions, and
context. Then, the framework finds whatever is still broken, unclosed, or just
"looks good" but covers a void. Next, it fixes the problem, checks the effect,
and repeats the loop until nothing important is left.
This is not a promise of a "smarter model." It is the promise of a process that
converges to quality.

confidently ship it.

## The Origin Story

The history of its creation isn't typical.
It started with two veterinarians who wanted to use AI to build a tool supporting other
veterinarians. They were not programmers. They had vision, stubbornness, and a belief that AI
could be used to build a product independently. At first, it worked. The code was written. The
application grew. Then came the second phase: the models began getting lost in their own code,
adding unnecessary layers, and "fixing" things that weren't broken.
Then, instead of messaging the AI "build this," a new question emerged:

> ### **"What is still wrong?"**

From this frustration, a new prompt wasn't born. A methodology was.
And from that methodology - this framework.

Underneath, ݆ ݆𝚅𝚒𝚋𝚎𝚌𝚛𝚊𝚏𝚝𝚎𝚍. ties together several key layers:

- Tools for reading the repository structure, so the agent doesn't guess the architecture.
- Intentions retrival system, so it doesn't start from scratch every session.
- Follow-up and Marbles loops, which systematically reduce errors.
- Multi-agent orchestration, while one perspective is rarely enough.
- The shipping layer, which guards not just the code, but the product's readiness for
  real-world use.

This means ݆ ݆𝚅𝚒𝚋𝚎𝚌𝚛𝚊𝚏𝚝𝚎𝚍. doesn't stop at "it works on my machine."

It keeps asking:

- Can it be found?
- Can it be understood?
- Can it be installed?
- Can someone trust it?
- Can it actually be shipped?

That is what separates working code from a finished product.
