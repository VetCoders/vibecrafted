Generowanie kodu jest stochastyczne. Pętla Marbles używa priority-weighted denoising loops, żeby bezlitośnie redukować wariancję i doprowadzić system do stanu gotowego do wdrożenia.

Twórz

Rozrysuj architekturę, zainicjuj kontekst i uruchom workflow ERI. Wymuś na agentach szacunek do AST i historii.

Domykaj

Systematycznie redukuj entropię pętlami Marbles, aż wszystkie problemy P0, P1 i P2 zostaną domknięte.

Wdrażaj

Przeaudytuj powierzchnię produktu przez DoU, dopoleruj przez Decorate i domknij przez Hydrate przed finalnym Release.

Marbles

Odszumiaj, aż okrąg się domknie

Agenci AI to stochastyczne papugi. Każda generacja dokłada szum. Tego nie naprawisz promptem. Iterujesz.

Izomorfizm

Image diffusion:  noise → denoise × N → image
Code diffusion:   chaos → reduce_entropy × N → product

Krok 0: Czysty szum (bez kontekstu, bez struktury)
Krok 1: Init — wyłaniają się grube kształty
Krok 2: Implement — generacja detalu
Krok N: Domknięte — DoU < threshold = DoD

Nie ma stałej liczby pętli. Pętla działa, aż P0=P1=P2=0 — okrąg jest pełny. Wczesne pętle robią duże korekty strukturalne; późne robią drobne dopracowania.

DoU

Definition of Undone

Agenci AI rozwiązali najtrudniejszą część tworzenia oprogramowania: sprawienie, że rzecz działa. Nie rozwiązali jeszcze drugiej najtrudniejszej: sprawienia, że da się do niej dotrzeć.

Luka rzadko dotyczy jakości kodu. To brutalna przestrzeń między działającym systemem a produktem, który obcy mogą odkryć, zainstalować, zrozumieć, zaufać mu i kupić.

Diagnoza Foundera:

  • + Testy przechodzą
  • + Architektura jest zdrowa
  • + README istnieje
  • – Nikt nie może tego znaleźć z Google
  • – Nikt nie może tego zainstalować bez toolchainu
  • – Nikt nie może za to zapłacić

Punkty 4–6 to właśnie Definition of Undone. Faza 3 wypycha system w stronę rynku.

Referencje

Źródła, wpływy i punkty odniesienia

Bibliografia poniżej rozszerza listę z manifestu o wpływy z obszaru modelowania strukturalnego i konwergencji, które wracają także w wewnętrznej pracy AICX nad groundingiem.

  1. Morris, K.; Böckeler, B. (2026). Humans and Agents in Software Engineering Loops; Harness Engineering; Context Engineering for Coding Agents. Thoughtworks / Martin Fowler.

    Ramowanie why-loop / how-loop, projekt harnessu i dyscyplina workflow prowadzonego przez człowieka, które stoją za SHACE i rolą operatora.

  2. Gong, L.; Elhoushi, M.; Cheung, A. (2024). AST-T5: Structure-Aware Pretraining for Code Generation.

    Wątek structure-aware pretraining, który wzmacnia upór 𝚅𝚒𝚋𝚎𝚌𝚛𝚊𝚏𝚝𝚎𝚍., że modele nie powinny pisać do repo na ślepo.

  3. Guo, D. et al. (2021). GraphCodeBERT: Pre-training Code Representations with Data Flow.

    Komplementarny wynik z obszaru structure-aware learning, który wzmacnia argument za wyprowadzaniem kształtu kodu na zewnątrz, mimo że GraphCodeBERT modeluje data flow, a nie AST.

    arXiv

  4. Cunningham, W. (1992) and later technical debt literature.

    Konceptualny przodek DoU: działający kod nie jest jeszcze tym samym co zdrowy, godny zaufania i gotowy do wdrożenia system.

  5. CEGIS / counterexample-guided synthesis and later counterexample-guided optimization literature.

    Linia rodowa konwergencji stojąca za Marbles: zapętlaj, oglądaj porażkę, dodawaj counterexamples, redukuj przestrzeń możliwych programów i powtarzaj, aż powierzchnia przestanie kłamać.

    Counterexample-guided synthesis overview

  6. VetCoders (2026). The𝚅𝚒𝚋𝚎𝚌𝚛𝚊𝚏𝚝𝚎𝚍. Manifesto.

    Główne źródło dla SHACE, Marbles, Loctree Mapping, PSCD i founder-grade Definition of Undone jako oryginalnych koncepcji𝚅𝚒𝚋𝚎𝚌𝚛𝚊𝚏𝚝𝚎𝚍..
